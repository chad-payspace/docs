Real-time Event Messages

These template based event messages enable you to display data in a presentable way to your customers in real-time.

Methods for consuming Real-time Event Messages

  1. Connect to our real-time event messages socket server
  2. Render an iFrame on your checkout page
  3. Event webhooks delivered to the ntf_url for each transaction

Connect to our event webhook socket server

Our socket server will publish all Real-time Event Messages notifications via a web socket for your front end client to consume. You will authenticate to the socket server using a public token that will be provided.

Getting Started

To use this service, make a call to the URL with the necessary parameters to validate and return transaction updates.

Service URL

http://socket.payper.ca?key=<PublicKey>&id=<txn>

URL Components

  • Server: http://socket.payper.ca
    • The server to which the call is made.
  • PublicKey: Unique user password provided by the administrator.
    • Example: 56516523656323662363
  • txn: Unique transaction identifier.

All parameters are required. The TXN and the PUBLIC KEY are validated by the system.

Calling Method

The call to the server is made through SOCKET.IO and returns a response where you can find different formats and information on the status of the transaction.

Parameters

ParameterTypeExample
ServerTexthttp://socket.payper.ca
PublicKeyTextNGzDvDoAhrJbyAkGC4367
txnNumeric56516523656323662333363

Usage Example

To receive real-time notifications for a specific transaction, you can use the following URL:

http://socket.payper.ca?key=NGzDvDoAhrJbyAkGC4367&id=56516523656323662333363

Render an iFrame on your checkout page

Getting Started

To use this service, make a call to the URL with the necessary parameters to validate and return transaction updates.

Service URL

https://socket.payper.ca/<TXN>/<Amount>/<Format>

URL Components

  • Server: http://socket.payper.ca
    • The server to which the call is made.
  • TXN: Unique transaction identifier.
    • Example: 56516523656323662363
  • Amount: Transaction amount.
    • Example: 1.0
  • Format: The format in which you want the iFrame to display the data.
    • Example: html

All parameters are required. The TXN and Amount must match the actual transaction information as they are validated by the server.

Calling Method

The call to the server is made using the GET protocol and returns an HTML page with an iFrame. This iFrame contains a WebSocket connection that is responsible for receiving messages in the specified format.

Parameters

ParameterTypeExample
ServerTexthttp://socket.payper.ca
TXNNumeric56516523656323662363
AmountFloat9.99
Formattext/html/jshtml

Usage Example

To receive real-time notifications for a specific transaction, you can use the following URL:

http://socket.payper.ca/56516523656323662363/1.0/html

This link will return an HTML page with an iFrame for the transaction with ID 56516523656323662363 and amount 1.0, displaying the data in HTML format.

Event webhooks delivered to the ntf_url for each transaction

  • Status Webhooks are sent out when the status of a transaction changes, however, Event Webhooks can be sent out at any time during a transaction's life
  • This type of webhook is not used to mark transaction statuses, but rather to pass along information from Paybilt to the customer about their transaction
  • When you receive an Event Webhook you can use the "event_webhook_data" field of the JSON object returned to access three option fields: 'text', 'html', and 'js_enabled_html'. Use these to display the content in your chosen format to your customers

Displaying an Event Webhook

In the above example, the configuration for the Event Webhook was set to render the HTML for transactions stuck on status: payment, action: pending, longer than expected. In this case the Event Webhook triggers and renders the HTML.

For this functionality we integrated Socket.IO to listen for the Event Webhooks in order for them to display at the correct time.

Below shows an example of how you can integrate Socket.IO for this purpose. In this example we are putting the socket server into our Python Flask app. After setting up a socket server with your language of choice you will want to set up an endpoint that will listen for the Event Webhooks. You can set the notification URL for the transaction to be this '/listener' endpoint.

# python socket server endpoint 

@app.route('/listener', methods=["POST"], strict_slashes=False)
def listen():
    request_data = request.get_json()
    pb = dict(request_data)
    if pb.get('is_event_webhook'):
        udfs = pb.get('udfs')
        sio.emit(event='event_webhook', data=(pb.get('event_webhook_data')), to=udfs[0])

Then on the front end you would have your socket listen to this 'event_webhook' emit event. Upon connecting to the server a unique ID will be generated for the socket on the front end. We use this socket ID as the transaction udf so that on the server side we can use this udf to send to that specific socket. This way the information is not emitted to every socket connected to your server, just the socket on the page that transaction was created on.

# connect to the socket server when the page first loads
  so before the transaction is created 

# set the udf to the socket id so later when the server emits the event webhook
  it only emits to the socket associated with the transaction created on that page

let udf
const sio = io.connect()
sio.on('connection', () => {
    udf = sio.id
})

# the below code snippet is later in the code, after clicking the button that starts
  the tranasction, the socket will now listen to the 'event_webhook' emit event

sio.on('event_webhook', (event_webhook) => {
  const eventWebhookDisplay = document.getElementById('event-webhook-display');
  eventWebhookDisplay.innerHTML = event_webhook
})

This is just one example of Event Webhook implementation, they do not need to be used with sockets and you can display the data to your customers however you best see fit.
